# 5.3 会话的生命周期——从创建到销毁

上一节我们了解了 WebSocket RPC 通信——前端和后端之间的"对讲机"是怎么工作的。但"对讲机"只是通信管道，真正承载对话内容的，是一个叫"会话"的东西。这一节我们就来聊聊：一个会话从出生到死亡，都经历了什么。

## 银行账户：一个会话的一生

你去银行办过业务吗？一个银行账户的一生大概是这样：先带身份证开户（创建），拿到卡和账号之后就可以用了（激活），然后存取款、转账、买理财（使用），如果哪天不想用了可以冻结（归档），万一后悔还能解冻，实在不要了就销户（销毁）。要是你和朋友想各管各的钱但共享一段流水记录，还可以从同一个账户"另存为"出一个新账户（分支）。

会话（Session）就像银行账户。它是用户和 Agent 之间一次持续对话的容器，里面装着所有消息、工具调用记录、权限状态、Token 消耗……每个会话绑定一个 Agent 后端（Claude 或 Pi），就像每个账户绑定一个客户经理。

## 会话的七个阶段

### 1. 创建——开户

用户点击"新建会话"，或者在 CLI 里执行 `session create`，系统就会为这个会话分配一个唯一的 ID（用 `randomUUID()` 生成），并在工作区目录下创建一个 `sessions/{id}/` 文件夹。此时的会话就像刚开好户的银行账户——有了账号，有了档案袋，但还没开始任何业务。

源码中 `createSession()` 方法负责这一步：它生成 ID、初始化默认配置（权限模式、工作目录、模型连接），然后把会话信息写入磁盘。写入磁盘？对，这就引出了下一个重要的概念。

### 2. 持久化——存折上的流水

银行账户的交易记录不会只存在柜员的脑子里——它们被一笔一笔记在存折上。会话也一样：每条消息、每次工具调用的结果，都以 JSONL（JSON Lines）格式逐行追加到 `session.jsonl` 文件中。

JSONL 的好处是"一行一记录"：第一行是会话的元数据（SessionHeader，包含名称、时间戳、Token 用量等），后面的每一行是一条消息（StoredMessage）。这种格式不需要把整个文件读进内存就能解析——加载会话列表时只读第一行就够了，需要查看历史消息时再逐行往下读。更重要的是，应用重启后会话不会丢失——从磁盘上读回来就行。

### 3. 使用——存取款

这是会话最活跃的阶段：用户发消息，Agent 处理并返回结果，循环往复。每一轮对话，Agent 都会通过工具调用读取文件、执行命令、访问 API，然后把这些结果和对话文本一起写入 JSONL。

会话在这个过程中有一个用户可控的**状态流转**，内置了五种状态：`todo`（待办）、`in-progress`（进行中）、`needs-review`（待审核）、`done`（完成）、`cancelled`（已取消）。这就像订单状态跟踪——你可以给每个会话标上当前进度，一目了然地知道哪些在忙、哪些在等、哪些做完了。

### 4. 命名——账户别名

新创建的会话默认没有名字，标题栏显示的是空白。当第一条用户消息发送后，系统会调用 Agent 的 `runMiniCompletion()` 方法——一个不带工具、不带系统提示的轻量级文本补全——让模型根据对话内容生成一个 2-5 个词的标题。这就是 `regenerateTitle()` 的工作：它从对话中抽取几条有代表性的用户消息（开头、中间、末尾各一条），拼成一段提示词，让模型起个名字。

当然，你随时可以手动改名（`renameSession()`），就像给银行账户加个备注名一样方便。

### 5. 归档——冻结账户

有些会话用完了，但又不想删——也许以后还要回顾。这时可以把它归档（Archive）。归档后的会话从主列表中隐藏，不再参与未读计数，但数据完好无损。如果哪天想起来了，还可以取消归档（`unarchiveSession()`），让它重新出现在主列表里。

源码中归档操作很简单：设置 `isArchived = true` 和 `archivedAt` 时间戳，然后持久化到磁盘并广播 `session_archived` 事件给前端。

### 6. 分支——另存为

这是会话生命周期中最有意思的部分。有时候你在对话中走到了一个岔路口：想试试另一种方案，但不想丢失当前的进展。怎么办？分支（Branch）——就像文档编辑器里的"另存为"，原文件不变，新文件从当前版本开始独立演化。

分支的关键在于**上下文切断点**。每个分支会话记录了 `branchFromMessageId`——从哪条消息开始分叉。Agent 在构建上下文时，会严格不包含父会话中这条消息之后的内容。源码中有两种分支策略：`sdk-fork`（Provider 级别的原生分叉，Claude SDK 支持）和 `seeded-fresh-session`（新建一个干净的会话，把截止到分支点的对话记录"播种"进去）。

### 7. 销毁——销户

当一个会话彻底不再需要时，可以删除它。`deleteSession()` 会先强制中止正在进行的 Agent 处理，清理 MCP 连接池、浏览器资源等所有关联状态，然后从内存中移除，最后删除磁盘上的整个会话目录——`sessions/{id}/` 文件夹及其中的所有文件（消息、附件、计划、下载数据）都会被永久移除。

这一步是不可逆的，就像银行销户——一旦执行，流水记录、账户余额，统统消失。

## SessionManager——银行经理

如果说每个会话是一个账户，那 SessionManager 就是管理所有账户的银行经理。它是整个会话生命周期的大管家，负责：

- **增删改查**：创建、删除、查找、重命名会话
- **状态流转**：更新会话的 todo/in-progress/needs-review/done 状态
- **消息分发**：把用户消息交给 Agent 处理，把 Agent 的响应推送给前端
- **工具执行协调**：管理 MCP 连接池、权限请求、凭证认证等
- **事件广播**：通过 WebSocket 向前端推送会话变更事件

SessionManager 在内存中维护一个 `Map<string, ManagedSession>`——所有活跃会话的索引。每个 ManagedSession 不仅包含会话配置，还持有对应的 Agent 实例、MCP 客户端池、Token 刷新管理器等运行时资源。

## 标签系统——给账户贴标签

除了状态，会话还支持标签（Label）。你可以给会话贴上"紧急""bug""feature"等标签，方便分类和筛选。标签系统甚至支持自动化：`evaluateAutoLabels()` 可以根据会话内容自动匹配规则并打标签，比如检测到对话中提到了某个关键词就自动标记。这就像银行的客户画像系统——根据交易特征自动打标签，帮你快速找到想要的那类客户。

## 会话间通信——跨账户转账

一个会话不是孤岛。通过 `send_agent_message` 工具，一个会话可以向另一个会话发消息。发送方会在消息前加上"我是谁"的信封信息，接收方则被告知可以用 `send_agent_message` 回复。这就像银行之间的跨行转账——两个独立账户，通过一条约定好的通道传递价值。

---

从开户到销户，从单账户管理到跨账户通信——会话的生命周期构成了 Craft Agents 运行时的骨架。每一个你看到的对话、每一次工具调用、每一条消息记录，都在这个骨架上有条不紊地流转。但这个骨架是怎么搭建起来的？服务器是怎么从零开始组装出 SessionManager、Agent 后端、MCP 连接池这些组件的？下一节我们来看 Server Builder——服务器组装流水线。
