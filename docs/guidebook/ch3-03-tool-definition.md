# 3.3 一个工具长什么样？工具定义解密

上一节我们认识了三类工具——内置工具、会话工具和外挂工具。但你有没有想过：AI 到底怎么"读懂"一个工具？它又不是人，不能看一眼就懂你的意图。事实上，每个工具都有一份结构化的"简历"，AI 靠这份简历来决定什么时候用、怎么用这个工具。

这份"简历"长什么样？我们用一个更生活化的比喻来说——**菜谱**。

## 工具就是一道菜谱

想象你在翻一本菜谱App。每道菜的信息都包含这几样东西：菜名（让你找到它）、菜品介绍（让你知道它是什么口味）、食材清单（告诉你需要什么原料）、烹饪步骤（告诉你怎么做）、特殊标注（比如"素食""快手菜"的标签）。

一个工具的定义，和菜谱如出一辙，也由五个要素组成。

### 1. name（名称）——菜名

就像"宫保鸡丁""番茄炒蛋"一样，每个工具都有一个名字。AI 通过这个名字来识别和调用工具。你可能已经见过一些：

- `Bash`——执行命令行的工具
- `Read`——读取文件的工具
- `mcp__linear__createIssue`——通过 MCP 连接的 Linear 服务里"创建 Issue"的工具

注意最后那个名字：`mcp__linear__createIssue`。这种 `mcp__服务名__工具名` 的命名方式是 MCP 工具的惯例——一眼就能看出它是哪个外挂服务提供的。

### 2. description（描述）——菜品介绍

光有菜名不够。"红烧肉"三个字，北方人想到的是五花肉，上海人想到的是偏甜的做法。描述的作用就是消除歧义，告诉 AI：这个工具到底能干什么、什么时候该用它。

比如 `Read` 工具的描述会写"读取本地文件系统中的文件内容"——这样 AI 就知道，当用户说"帮我看看 config.json 里写了什么"时，应该用这个工具，而不是 `Bash` 去跑 `cat` 命令。

好的描述是工具被正确使用的关键。描述写得太模糊，AI 可能用错工具；写得太啰嗦，又浪费宝贵的上下文空间。

### 3. schema（参数模式）——食材清单

食材清单规定了做这道菜需要什么原料、每种原料是什么类型（蔬菜还是调味料）、能不能省略。Schema（模式/结构定义）做的是同样的事：定义工具需要什么参数、参数是什么类型、是不是必填。

看一个真实的例子——API 工具的 Schema：

```typescript
{
  path: z.string().describe('API endpoint path, e.g., "/search"'),
  method: z.enum(['GET', 'POST', 'PUT', 'DELETE', 'PATCH'])
         .describe('HTTP method'),
  params: z.record(z.string(), z.unknown()).optional()
         .describe('Request body or query parameters'),
  _intent: z.string().optional()
         .describe('Describe what you are trying to accomplish'),
}
```

这里每个参数都用 `z.string()`、`z.enum()` 等方式声明了类型，`.optional()` 表示可省略，`.describe()` 给每个参数加了说明。这些 `z.xxx()` 是什么呢？

**Zod 是什么？** Zod 是一个 TypeScript 参数校验库，你可以把它理解为一个**质检员**——它确保食材清单上的每样东西都符合规格。`z.string()` 要求参数必须是字符串，`z.enum([...])` 限定只能是列表中的值，`.optional()` 表示这道食材可以不放。如果 AI 传了不符合规格的参数，Zod 会在执行前就拦住，避免"把盐当糖放"的灾难。

### 4. handler（处理函数）——烹饪步骤

食材备好了，接下来是烹饪。Handler（处理函数）就是工具的核心逻辑：收到参数之后具体怎么执行。它是一个异步函数，接收参数，干活，返回结果。

### 5. options（选项）——特殊标注

菜谱上常有些标签，比如"素食""快手菜""免烤箱"。工具也有类似的标注，通过 `annotations`（注解）来表达。最常见的一个是 `readOnlyHint`：

- `readOnlyHint: true`——这个工具只读不改，不会产生副作用
- `readOnlyHint: false`——这个工具可能会修改文件、执行命令，需要谨慎

AI 在决定是否自动执行工具时，会参考这些标注。只读工具通常可以放心执行，有副作用的工具则可能需要用户确认。

## 把五要素拼在一起

用代码来看，一个工具的完整定义大概长这样：

```typescript
tool(
  'Bash',                                    // 名称：菜名
  'Execute a shell command',                 // 描述：菜品介绍
  { command: z.string(), ... },              // Schema：食材清单
  async (args) => { /* 执行命令 */ },        // Handler：烹饪步骤
  { annotations: { readOnlyHint: false } }   // 选项：特殊标注
)
```

这五行代码就是一份完整的菜谱。AI 看到它，就知道：这个工具叫 Bash，能执行命令行，需要一个 `command` 参数，执行后可能有副作用。

## 上菜时的"质检标签"：返回值格式

菜做好了，端上桌时还会附上一张小标签——这道菜做成功了吗？工具的返回值也有类似的格式：

```typescript
{
  content: [{ type: 'text', text: '命令执行结果...' }],
  isError: true   // 可选，表示"做坏了"
}
```

`content` 是实际返回的内容，`isError` 是一个可选的标签。正常执行时不带这个字段；如果执行出错了，就加上 `isError: true`，就像上菜时贴一张"此菜翻车"的警告标签，让 AI 知道结果不可靠，需要换策略。

## 外挂工具的"自动翻译"：Schema 自动转换

你可能注意到一个问题：前面说内置工具和会话工具的 Schema 都是用 Zod 写的，但 MCP 外挂工具来自外部服务，它们提供的参数定义是 JSON Schema 格式——两种"语言"不一样，怎么办？

答案是自动翻译。框架内置了一个叫 `jsonSchemaToZodShape` 的函数，它会把 MCP 服务提供的 JSON Schema 自动转换成 Zod 格式。你不需要手写任何转换代码，外挂工具接进来的时候，它的参数定义就被自动"翻译"成框架能理解的样子了。

```typescript
// MCP 工具的 Schema 自动转换——无需手写
{
  ...jsonSchemaToZodShape(mcpTool.inputSchema),
  ...z.object({}).catchall(z.unknown()).shape,
}
```

这就好比你请了一位外国厨师，他带来的菜谱是英文的，但厨房里有个自动翻译机，秒速转成中文——你什么都不用管，照着做就行。

---

所以，一个工具的定义其实很简单：五个要素凑齐，AI 就能"读懂"这个工具、正确地使用它。而 MCP 外挂工具的加入，则被自动翻译悄悄搞定了。但外挂装备到底是怎么连进来的？它们和主程序之间怎么通信？下一节我们聊聊 MCP 协议——Agent 世界的"万能转接头"。
