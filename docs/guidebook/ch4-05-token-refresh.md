# 4.5 OAuth Token 刷新——认证的生命周期

上一节我们了解了模型解析——系统怎么知道一个模型 ID 对应哪种 Provider、需要哪些参数。但还有一个问题没聊：解析出模型之后，拿什么凭证去调用它？如果用的是 OAuth（开放授权）方式登录，那凭证可不是一把永久的钥匙，而是一张会过期的签证。

## 签证续签：一个关于过期的比喻

你去过国外吗？出国需要签证，签证有有效期——可能三个月，可能一年。到期之前你得续签，否则就非法滞留了。OAuth Token（开放授权令牌）就是这样一张"签证"：它让你有权访问某个服务（比如 Google 邮箱、Slack 工作区），但它会过期。过期了怎么办？续签。

那为什么不直接给一张永久签证呢？因为安全。你的签证只允许你在特定时间内、做特定的事情，一旦泄露，损失也是有限的——就像你给朋友一把临时钥匙，只能开客厅门，不能开卧室门，而且钥匙三天后就失效了。这就是 OAuth 的设计哲学。

## 什么是 OAuth？

OAuth（开放授权）是一种让用户授权第三方应用访问自己数据的协议。你在很多网站上见过"使用 Google 登录"按钮吧？点击之后，网站不会直接拿到你的 Google 密码，而是跳转到 Google 的页面，你同意授权后，Google 才给网站发一个 Token（令牌）。这个 Token 限制了网站能访问哪些数据、能访问多久。

打个比方：你给朋友一把临时钥匙——只能开客厅门，不能开卧室门，而且钥匙有使用期限。OAuth 就是那套"发钥匙"的规矩。

## Token 的生命周期

OAuth Token 的一生，和签证的流程几乎一模一样：

**1. 获取**——第一次登录时，系统用 OAuth 流程获取两种令牌：Access Token（访问令牌）和 Refresh Token（刷新令牌）。Access Token 是你每次调用 API 时出示的"签证"，Refresh Token 是你用来续签的"申请表"——平时用不到，但签证快到期时，凭它换一张新签证。

**2. 使用**——每次调用 API 时，请求头上带上 Access Token，就像出境时出示签证。服务器验证 Token 有效，就放行。

**3. 过期**——Token 有有效期，通常 1 小时左右。过期后，API 返回 401（未授权）错误，就像边检官看了你过期的签证，摇摇头说："不行，先去续签。"

**4. 刷新**——用 Refresh Token 向认证服务器请求新的 Access Token，就像拿申请表去签证中心续签。刷新成功后，你拿到了新签证，又可以继续出行了。

这个"获取-使用-过期-刷新"的循环，就是 Token 的生命周期。

## Craft Agents 的自动刷新机制

好，理论讲完了。现在来看 Craft Agents 是怎么做的——关键在于：**你完全不需要手动续签**。

整个过程是这样的：

1. 子进程（Pi Agent Server）调用 API 时，Access Token 过期了，API 返回 401 错误
2. 子进程通过 JSONL 协议发一条 `error` 消息给主进程，消息里包含 401 或 421 状态码
3. 主进程检测到这是认证错误，自动调用 `refreshAndPushTokens()` 刷新逻辑
4. 刷新成功后，主进程通过 `token_update` 消息把新的 Access Token 推送给子进程
5. 子进程用新 Token 重试刚才失败的请求

你在界面上看到的，只是对话稍微停顿了一下，然后继续正常运行。续签在你不知情的时候已经完成了——就像你在候机厅打了个盹，醒来发现签证已经续好了，航班也没有耽误。

## 竞态保护：不能同时续签

如果你只有一个 PiAgent 实例，上面的流程毫无问题。但现实中，Craft Agents 可能同时运行多个 PiAgent 实例，而它们可能共享同一个 OAuth 连接——比如两个对话窗口都用同一个 GitHub Copilot 账号。

这时候就有一个隐患：如果两个实例几乎同时检测到 Token 过期，都跑去刷新，会怎样？

答案是：可能出大问题。因为某些 Provider（比如 Microsoft）在刷新时会轮换 Refresh Token——旧的 Refresh Token 刷新后就失效了，新的才会生效。如果两个实例同时刷新，第一个拿到的新 Refresh Token 可能被第二个的刷新操作覆盖，导致第一个拿到的 Token 失效。这就像两个人同时去银行窗口办同一个账户的业务——窗口只能一个一个来，否则账户余额就乱了。

Craft Agents 的解决方案是**全局互斥锁（Mutex）**。`PiAgent` 类有一个静态变量 `globalRefreshMutex`，它是一个 `Map`，以连接标识为 key、以刷新 Promise 为 value。当某个实例准备刷新时：

- 先检查 `globalRefreshMutex` 里有没有同连接标识的刷新正在进行
- **有**：不启动新的刷新，等待已有的刷新完成，然后从凭证存储中拿到新 Token，推送给自己的子进程
- **没有**：启动刷新，把 Promise 存入 `globalRefreshMutex`，其他实例看到它就会等待

就像银行窗口的叫号系统——同一时间只办一个人的业务，其他人排队等着，办完了再叫下一个。代码里只有几行，但解决的可是实打实的并发问题。

## 不同的认证类型

最后，来区分一下 Craft Agents 里常见的几种认证方式。它们的"保质期"各不相同：

- **API Key（API 密钥）**：永不过期，像身份证——办一次，终身有效（除非你自己吊销它）。适用于 OpenAI、Anthropic 这类直接购买 API 服务的场景。
- **OAuth Token（开放授权令牌）**：会过期，像签证——有有效期，到期前必须续签。适用于 ChatGPT Plus、GitHub Copilot、Google、Slack 等通过 OAuth 授权的服务。
- **IAM 临时凭证（AWS 身份访问管理）**：临时通行证——有效期更短（几分钟到几小时），由 AWS STS 服务动态签发，过期后需要重新获取。适用于 AWS Bedrock 这类云服务。

为什么要有这些区别？本质上是安全性和便利性的权衡。API Key 简单好用，但一旦泄露就麻烦了；OAuth Token 过期了会自动续签，泄露的损害有上限；IAM 凭证最短命，安全性最高，但管理也最复杂。Craft Agents 支持所有这几种方式，并且针对 OAuth Token 提供了自动刷新——你只需要在第一次登录时授权，后续的续签全由系统搞定。
