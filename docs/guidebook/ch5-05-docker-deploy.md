# 5.5 Docker 部署——容器化的 Agent

上一节我们了解了服务器构建器，知道了怎么把 Craft Agents 打包成一个可执行的服务端程序。但构建只是第一步——你手里有了这个程序，接下来要把它部署到服务器上跑起来。怎么确保开发机上测好的环境，到服务器上不出幺蛾子？怎么让部署像拧钥匙一样简单？答案就是 Docker。

想象你要把一批家具从北京运到上海。如果每件家具都裸着搬上火车，到了上海再一件件摆好，不仅费劲，还容易磕碰。但如果你把所有家具装进一个标准集装箱，卡车拉到火车站，火车运到上海，上海再用卡车拉走——全程不拆箱，到地方一开门就能用。Docker（容器平台）就是软件世界的集装箱：它把应用和所有依赖打包成一个 Container（容器），不管运到哪台机器上，打开就能跑。

## 什么是 Docker？

Docker 是一个容器化平台，它的核心理念就是把应用"装进箱子"里运行。这个箱子里有完整的运行环境——操作系统依赖、运行时、配置文件，一应俱全。箱子外面是宿主机（Host），箱子里是容器（Container），两者互不干扰。

你可以把容器理解成一个轻量级的虚拟机，但又不完全是。虚拟机要模拟整套硬件、跑一个完整的操作系统，开销很大；容器共享宿主机的内核，只隔离应用层面，所以启动快、占用少。就像合租公寓和独栋别墅的区别——合租共享厨房和卫生间（内核），但每个人有自己的卧室（容器）；独栋别墅什么都得自己来（虚拟机），成本高得多。

## 为什么用 Docker 部署 Agent？

你可能会想：上一节不是已经能直接跑服务端程序了吗？为什么还要多一层 Docker？原因很实际：

### 1. 环境一致

"在我电脑上能跑"是开发界最著名的遗言。本地开发用的是 macOS，服务器跑的是 Ubuntu；本地 Bun 版本是 1.3，服务器上是 1.2——任何一点差异都可能让程序崩溃。Docker 把整个运行环境打包进镜像（Image），开发机和服务器用的是同一个镜像，环境完全一致，不会再有"在我电脑上能跑"的尴尬。

### 2. 一键部署

没有 Docker 的时候，部署一台服务器要装运行时、装依赖、配环境变量、设端口……步骤多，容易漏。有了 Docker，一个 `docker run` 命令启动整个服务器，所有配置都在命令里写好了。不需要 SSH 上去装东西，不需要手动编辑配置文件，一行命令，服务就跑起来了。

### 3. 隔离安全

容器有自己的文件系统和网络空间，和宿主机互相隔离。Agent 在容器里执行任务、读写文件，宿主机的系统不受影响。就算容器里的程序出了问题——占满内存、疯狂写磁盘——也只会在容器内部折腾，不会把宿主机搞崩。

### 4. 易于扩展

需要更多算力？不需要重新配置环境，多起几个容器就行。Docker 的启动速度是秒级的，横向扩展非常方便。配合编排工具（比如 Docker Compose、Kubernetes），可以做到自动扩缩容——流量大的时候多起几个容器扛住，流量小的时候缩减容器节省资源。

## Docker 部署的命令

说了这么多好处，来看实际操作。用 Docker 部署 Craft Agents 服务器，核心就是一条命令：

```bash
docker run -d \
  -p 9100:9100 \
  -e CRAFT_SERVER_TOKEN=<token> \
  -e CRAFT_RPC_HOST=0.0.0.0 \
  -v craft-data:/home/craftagents/.craft-agent \
  craft-agent-server
```

逐行拆解：

- **`docker run`**——启动一个容器。这是最核心的命令，后面所有参数都是在告诉 Docker "这个容器怎么配"。
- **`-d`**——后台运行（Detached 模式）。加上这个参数，容器启动后不会占据你的终端，你可以继续干别的事。不加的话，容器日志会直接打印在当前终端上——调试的时候有用，生产环境就不需要了。
- **`-p 9100:9100`**——Port Mapping（端口映射）。冒号左边是宿主机的端口，右边是容器内的端口。这里的意思是：把宿主机的 9100 端口映射到容器的 9100 端口。外部访问宿主机的 9100 端口，请求会自动转发到容器内部。就像酒店的房间号——你找 9100 房间（宿主机端口），前台会把你领到对应的 9100 号套房（容器端口）。
- **`-e CRAFT_SERVER_TOKEN=<token>`**——设置环境变量。这里是认证令牌，客户端连接服务器时必须提供这个 Token，否则连不上。把它想象成集装箱的密码锁——没有密码，谁也打不开。
- **`-e CRAFT_RPC_HOST=0.0.0.0`**——让服务器监听所有网络接口，允许外部连接。默认情况下，服务器只监听本地回环地址（127.0.0.1），外部设备连不进来。设成 0.0.0.0 就等于对所有人敞开大门——当然，有 Token 这把锁守着，不是谁都能进。
- **`-v craft-data:/home/craftagents/.craft-agent`**——Volume（数据卷）挂载。冒号左边是宿主机上的一个命名卷（Named Volume），右边是容器内的数据目录。容器内的 Agent 数据——会话历史、配置文件、MCP 连接信息——都存在这个目录下。挂载之后，即使容器被删除或重建，数据依然安全地保存在宿主机上。就像集装箱里放了一个保险柜，集装箱可以换，保险柜里的东西不会丢。
- **`craft-agent-server`**——镜像名称。这是上一节构建出来的 Docker 镜像，包含了运行 Craft Agents 服务器所需的一切。

## TLS 加密：给集装箱上锁

如果你只是在本机测试，`ws://` 连接就够用了。但一旦把服务器部署到公网，Token 明文在网上传输就非常危险——任何人只要截获了网络包，就能拿到你的 Token，冒充你连接服务器。

TLS（Transport Layer Security，传输层安全）就是解决这个问题的。它把 `ws://` 升级为 `wss://`——就像给集装箱加了一把锁，传输的数据全被加密，中间人截获了也看不懂。

启用 TLS 需要两步：

**第一步：挂载证书文件**

```bash
-v /path/to/certs:/certs:ro
```

把宿主机上的证书目录挂载到容器内的 `/certs` 路径，`:ro` 表示只读（Read-Only），容器只能读取证书，不能修改。证书是你的锁，锁不能被别人改。

**第二步：设置证书环境变量**

```bash
-e CRAFT_RPC_TLS_CERT=/certs/cert.pem \
-e CRAFT_RPC_TLS_KEY=/certs/key.pem
```

告诉服务器证书和私钥的位置。服务器检测到这两个环境变量后，会自动从 `ws://` 切换到 `wss://`，所有通信都经过加密。

完整带 TLS 的启动命令：

```bash
docker run -d \
  -p 9100:9100 \
  -e CRAFT_SERVER_TOKEN=<token> \
  -e CRAFT_RPC_HOST=0.0.0.0 \
  -e CRAFT_RPC_TLS_CERT=/certs/cert.pem \
  -e CRAFT_RPC_TLS_KEY=/certs/key.pem \
  -v /path/to/certs:/certs:ro \
  -v craft-data:/home/craftagents/.craft-agent \
  craft-agent-server
```

## 数据持久化：别让数据随容器而去

Docker 容器是临时的。你删掉一个容器，它里面的所有数据都会消失——会话记录、Agent 配置、MCP 连接凭证，全没了。这就像住酒店，退房的时候房间会被清空，你留下的东西都会被扔掉。

`-v` 参数就是解决这个问题的。它把容器内的数据目录映射到宿主机的一个持久化存储上。容器是临时的，但 Volume（数据卷）是持久的——容器删了可以重建，Volume 里的数据还在。

在上面的命令中，`craft-data:/home/craftagents/.craft-agent` 做的就是这件事：容器内 Agent 的所有数据都存在宿主机的 `craft-data` 卷里。重建容器时，只要挂载同一个卷，所有数据都还在——就像酒店虽然换了房间，但你的行李一直存在保险柜里，随时可以取回。

## 安全注意事项

最后，聊几点生产环境的安全常识：

**不要把 Token 提交到代码仓库。** Token 是服务器的钥匙，泄露了等于把家门钥匙公开贴在小区公告栏上。正确做法是通过环境变量或密钥管理服务注入，不要硬编码在代码或配置文件里。

**生产环境用可信 CA 证书。** 开发时可以用自签名证书测试，但生产环境务必使用受信任的 CA（Certificate Authority，证书颁发机构）签发的证书。自签名证书就像自己刻了个公章——你自己认识，别人不认；受信任 CA 的证书就像公安局盖的章，大家都认。

**最小权限原则。** 容器内尽量以非 root 用户运行。Craft Agents 的 Docker 镜像已经默认使用 `craftagents` 用户运行服务，不需要额外配置。如果你需要调整文件权限，可以用 `--user $(id -u):$(id -g)` 参数让容器以宿主机当前用户的身份运行，避免权限问题。
