# 4.2 JSONL 协议——两个进程怎么对话？

上一节我们了解了进程隔离的意义——子进程崩溃不会拖垮主进程，不同 AI 提供商的 SDK 也能各安其位。但一个新的问题立刻冒出来了：既然两个进程被隔在不同的"房间"里，它们之间怎么沟通？

## 传纸条的规矩

想象你和同事坐在两个隔音的房间里办公，不能面对面说话，也不能打电话。唯一的沟通方式是：把想说的话写在纸条上，从门缝底下塞过去。

这就是 JSONL over stdio（通过标准输入输出传输 JSON 行格式）做的事。主进程和子进程各有一扇"门"：主进程往子进程的 stdin（标准输入）塞纸条，子进程往自己的 stdout（标准输出）回纸条。每张纸条上写一行 JSON——这就是 JSONL（JSON Lines，JSON 行格式）。

JSONL 的规则简单到一句话就能说清：**一行就是一个完整的 JSON 对象**。不像普通 JSON 文件那样用方括号把所有内容包起来、读完整个文件才能解析，JSONL 是一行一封"短信"——读完一行就处理一行，不用等后面的内容。

```typescript
// 主进程写纸条
this.subprocess.stdin.write(JSON.stringify(cmd) + '\n');

// 子进程读纸条
this.readline.on('line', (line: string) => {
  const msg = JSON.parse(line);
  // 根据 msg.type 分发处理
});
```

这种方式不需要额外的网络端口，不需要复杂的序列化框架，只要进程还活着，纸条就能送达。

## 六步入职：一个新员工怎么开始干活？

子进程不是一启动就能接活的——它就像一个刚入职的新人，得先办完手续才能上岗。整个流程分六步：

**第一步：spawn（雇佣）**。主进程调用 `spawn()` 启动子进程，就像公司和新员工签了合同。此时人已经到了，但还没拿到工牌，什么活也干不了。

**第二步：init（入职培训）**。主进程通过 stdin 发送一条 `init` 消息，内容相当于一本工作手册：API Key（接口密钥）、模型名称、工作目录、会话路径、认证方式……所有干活需要的基础信息一次性交底。子进程收到后完成内部初始化，建立和 AI 模型的连接。

**第三步：ready（准备就绪）**。子进程初始化完成后，回一条 `ready` 消息，附带上一个 callbackPort（回调端口号）。这句"我准备好了"的意思是：手续办完了，随时可以接活。

**第四步：register_tools（分配工具）**。主进程告诉子进程"你可以用这些工具"——包括会话级工具（如 SubmitPlan、call_llm）和 MCP Source 工具等。工具的名称、参数格式、功能描述都通过这条消息发过去，让 AI 模型知道它有哪些"手"可以调用。

**第五步：prompt（分配任务）**。一切就绪，主进程发送 `prompt` 消息，包含用户的输入和系统提示词。新人正式开工。

**第六步：event（汇报进展）**。子进程开始工作后，会持续发送 `event` 消息，把 AI 的回复一段一段地"流"回来。你看到界面上文字逐字出现，背后就是一条条 event 消息在飞。

## 两张纸条清单：谁发给谁什么？

把这些消息按方向归类，就得到两张清单。

**入站消息（主进程发给子进程）**，相当于老板给外包团队的指令：

| 消息类型 | 用途 |
|---------|------|
| `init` | 入职培训，交付工作手册 |
| `prompt` | 分配任务，启动对话 |
| `register_tools` | 登记可用工具 |
| `abort` | 紧急叫停当前任务 |
| `steer` | 中途转向，插入新指令 |
| `compact` | 压缩上下文，释放 Token 空间 |
| `mini_completion` | 小型补全请求（如生成标题） |
| `llm_query` | 直接向模型发问 |
| `token_update` | 推送刷新后的认证令牌 |
| `shutdown` | 关闭子进程 |

**出站消息（子进程发给主进程）**，相当于外包团队给老板的汇报：

| 消息类型 | 用途 |
|---------|------|
| `ready` | 准备就绪，附回调端口号 |
| `event` | AI 回复的流式输出 |
| `pre_tool_use_request` | 请求权限检查"这个操作允许吗？" |
| `tool_execute_request` | 请求代为执行工具 |
| `session_id_update` | 更新会话标识 |
| `error` | 报告错误 |

## 出错了怎么办？

子进程遇到问题，会发一条 `error` 消息，带上错误信息和错误码。这不是"甩锅"——而是在向总部求援。

有一种错误特别值得关注：认证错误。当使用 OAuth 登录的方式（如 ChatGPT Plus、GitHub Copilot）时，Access Token（访问令牌）会过期。子进程发现请求被拒绝（收到 401 或 421 状态码），就会发 error 消息。主进程收到后，自动用 Refresh Token（刷新令牌）换取新的 Access Token，然后通过 `token_update` 消息推送给子进程——整个过程对你透明，不需要重新登录。

这里有个细节值得多说一句：多个 PiAgent 实例可能共享同一个连接。如果三个子进程同时发现令牌过期，都去刷新，就会造成 Race Condition（竞态条件）——三个刷新请求抢着跑，不仅浪费资源，还可能互相覆盖。代码里用了一个全局 Mutex（互斥锁）来解决：同一时刻，同一个连接只允许一个刷新操作在跑，后来的实例发现有人在刷新了，就乖乖等着，等别人刷完了直接用新令牌。
