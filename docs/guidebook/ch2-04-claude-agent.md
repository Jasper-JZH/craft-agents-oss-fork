# 2.4 ClaudeAgent——Claude SDK 在进程内直接工作

上一节我们了解了 BaseAgent 这套"毛坯房"——水电管道全铺好了，只等你装修。现在，让我们走进第一套已经装修好的房子：ClaudeAgent。

## 自带发动机的车

想象你去买车。有一种车，发动机已经装好了，油箱也加满了，你坐上去拧钥匙就能开。另一种车，发动机在隔壁仓库里，你得先打电话让人把发动机运过来、接上管线、调试半天，才能启动。如果你只是想赶紧上路，显然选第一种。

ClaudeAgent 就是那辆"自带发动机的车"。它使用 Anthropic 官方提供的 Claude Agent SDK，在同一个 Process（进程）里直接运行 AI 对话循环。不需要额外启动子进程，不需要跨进程通信，不需要中间协议——SDK 就在进程内，像发动机装在车上一样，直接驱动一切。

## 核心 API：query() —— 发动引擎的钥匙

ClaudeAgent 的核心对话能力全部围绕 SDK 的 `query()` 函数展开。每次用户发一条消息，就创建一次新的 `query()` 调用，就像每次打电话都要拨一次号。

`query()` 返回的是一个 AsyncIterable（异步可迭代对象）。别被这个名字吓到——它就像一条流水线：AI 的回复不是一口气全给你，而是一段一段地持续产出。你可以用 `for await ... of` 循环逐条接收，也可以在收到足够内容后随时停止。这种流式输出让你不用傻等 AI 写完几百行代码再一起显示，而是边生成边呈现，体验更流畅。

### resume —— 回拨上一次通话

每次 `query()` 都是新的一次调用，但对话往往是连续的。怎么办？SDK 提供了 `resume` 选项，让你可以在新的 `query()` 调用中延续之前的会话。这就像打电话时的"回拨"功能——上次聊到一半挂了，这次拨回去继续聊，上下文全都在。

### forkSession —— 另存为新分支

有时候你想在当前对话的基础上探索一个不同的方向，但又不想丢失原来的对话记录。`forkSession` 选项就是为此而生：它从当前会话复制一份出来，创建一个新分支。就像文档编辑里的"另存为"——原始文件不变，你在副本上自由修改。两个分支从此独立演进，互不干扰。

## 工具注册：三种"装备"的安装方式

Agent 光能聊天不行，还得能干活。ClaudeAgent 支持三种工具注册方式，就像一辆车可以选装三种不同来源的配件。

### 内置工具——出厂自带的随车工具箱

SDK 自带了 Claude Code 的预设工具集（Preset Tools），包括 Bash（执行命令）、Read（读取文件）、Write（写入文件）等常用工具。不需要自己实现，只需一行配置即可启用：

```typescript
tools: { type: 'preset', preset: 'claude_code' }
```

就像买车时附赠的随车工具箱——千斤顶、扳手、三角牌——基础维修全够用。

### MCP 工具——专业维修站的专业设备

内置工具覆盖了常见需求，但有时你需要更专业的能力。这时可以通过 MCP（Model Context Protocol）接入外部工具服务器。ClaudeAgent 使用 `createSdkMcpServer()` 在进程内创建一个代理服务器，把外部 MCP 工具桥接到 SDK 的对话循环中。

回到买车的比喻：内置工具是随车工具箱，MCP 工具就是开去专业维修站用的举升机和诊断仪——能力更强，但需要先"对接"才能用。

### 会话级工具——随车改装件

还有一种工具注册方式同样通过 MCP Server 实现，但它不是连接外部服务器，而是直接在主进程内注册并执行。这种工具在创建会话时动态添加，随着会话的结束而消失。就像你给车临时加装了一个手机支架——不改变车的结构，但用的时候很方便。

## 权限控制：流水线上的质检站

Agent 有了工具就能干活，但不是所有操作都该放行。删除文件、执行危险命令——这些需要把关。ClaudeAgent 的做法很巧妙：它不使用 SDK 自带的权限系统，而是通过 `bypassPermissions` 绕过，然后用 SDK 的 `hooks.PreToolUse` 回调（Hook）自己实现权限检查。

Hook（钩子）是什么？想象一条工厂流水线，每个产品经过每个工位时都会被检查。`PreToolUse` 钩子就是在"工具执行"这个工位前加了一个质检站——每个工具执行前都先过一遍你的自定义检查函数，通过才放行，不通过就拦截。

```typescript
hooks: {
  PreToolUse: async (toolName, input) => {
    // 你的权限检查逻辑
    return isAllowed ? 'allow' : 'deny';
  }
}
```

为什么不直接用 SDK 的权限系统？因为 Craft Agents 有自己的一套 PermissionManager（上一节毛坯房里的"门禁系统"），权限规则更丰富、和 UI 交互更深。用自定义钩子对接这套门禁，比改造 SDK 内置权限灵活得多。

## 认证注入：出发前加满油

要让 SDK 调用 Anthropic 的 API，必须提供认证信息——要么是 API Key，要么是 OAuth Token。ClaudeAgent 的认证注入方式非常直接：通过 `process.env` 设置 Environment Variable（环境变量），SDK 自动从中读取。

就像开车前给油箱加满油——油加好了，路上就不用操心动力来源。认证信息注入到环境变量后，SDK 在每次 API 请求时自动携带，ClaudeAgent 不需要手动管理 Token 的传递和刷新。

## 事件转换：翻译官

SDK 输出的事件格式是 Claude 自己的定义，而 Craft Agents 的上层代码统一使用 `AgentEvent` 格式。两者不一样，怎么办？ClaudeAgent 用一个 ClaudeEventAdapter（适配器）来做转换——它就像一位翻译官，把"法语"（SDK 的事件格式）翻译成"通用语言"（AgentEvent），让上层代码不用关心底层引擎说的是哪国话。

这种 Adapter（适配器）模式在软件设计里非常常见：当两个系统的接口不兼容时，中间加一层适配器做格式转换，双方都不用改自己的代码。

---

ClaudeAgent 就是这样一套"精装修"：在 BaseAgent 毛坯房的基础上，用 Claude Agent SDK 当发动机，用 `query()` 驱动对话，用三种方式注册工具，用自定义钩子做权限检查，用环境变量注入认证，用适配器统一事件格式。一切都在同一个进程内完成，简洁高效。

但不是所有人都想开"自带发动机的车"。另一种引擎 PiAgent 则选择了一个完全不同的工作方式——住在隔壁房间。
