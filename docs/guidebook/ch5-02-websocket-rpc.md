# 5.2 WebSocket RPC——客户端和服务器的对话

上一节我们了解了无头服务器——它没有界面，安静地跑在后台，等待指令。但一个只会低头干活的服务器没什么用，它得能和外界沟通才行。问题是：怎么沟通？

## 对讲机 vs 电话

想象两个场景。

场景一：你用手机给同事打电话。每说一件事，都要拨号、等接通、说完、挂断。想再说下一件？再拨一次。十件事，拨十次号。这就是 HTTP（超文本传输协议）的工作方式——每次请求都要建立新连接，拿到响应就断开。

场景二：你和对面的同事各拿一台对讲机。频道调好之后，你想说就说，对方随时能听到；对方说话你也随时能听到，不用再拨号。一次通话，持续到你们都按了"结束"键。这就是 WebSocket（网页套接字）的工作方式——建立一次连接，保持打开，双向实时通信。

Craft Agents 的客户端和服务器之间，用的就是"对讲机"。原因很简单：Agent 的对话是实时的——AI 的回复一个字一个字地流回来，服务器的事件（Event）随时可能发生，客户端可不想每隔半秒问一句"有新消息吗？"。WebSocket 天然适合这种场景。

## RPC：按遥控器，远处就动

但光有"对讲机"还不够。对讲机只解决了"怎么说"的问题，没解决"说什么"的问题。

你用过电视遥控器吧？按"音量+"，电视音量变大；按"换台"，频道切换。你不需要知道电视内部怎么实现的，只需要知道按哪个按钮。RPC（Remote Procedure Call，远程过程调用）就是软件世界的遥控器——客户端调用一个方法名，服务器执行对应的操作，然后把结果返回来。

把 WebSocket 和 RPC 组合在一起，就得到了 Craft Agents 的通信框架：**用对讲机按遥控器**。客户端通过 WebSocket 连接服务器，然后通过 RPC 方法名调用服务器上的功能，服务器的结果再通过同一条连接返回。

## 两台设备：WsRpcServer 和 WsRpcClient

这个框架有两个核心角色，定义在 `@craft-agent/server-core` 包中：

- **WsRpcServer**：服务器端。监听 WebSocket 连接，验证身份，把收到的 RPC 请求分发给对应的处理器（Handler），还负责主动向客户端推送事件。
- **WsRpcClient**：客户端。发起 WebSocket 连接，发送 RPC 请求，接收服务器推送的事件，断线后自动重连。

它们配合工作的流程就像对讲机的频道约定：客户端喊一个频道号，服务器就知道该找谁接话。

## 一次完整的连接：从敲门到通话

客户端和服务器建立连接，不是连上就能用的，得走一套流程：

**握手（Handshake）**：客户端连上服务器后，第一件事是发送握手消息，里面带着协议版本、工作区 ID、能力列表等信息。这就像进门刷卡——服务器验证你的身份，确认你说的是同一种"语言"，然后才让你进来。如果需要认证（比如远程连接时），客户端还要出示 Token（令牌），服务器验证通过才放行。

**心跳（Heartbeat）**：连接建立后，服务器每 30 秒发一次 ping。客户端收到后回复 pong，告诉服务器"我还在线"。如果连续 2 次没收到回复，服务器就认为客户端掉线了，关闭连接。这就像对讲机里的"收到请回答"——没人应答，就当对方不在了。

**请求和响应（Request/Response）**：客户端通过 `invoke()` 调用服务器上的 RPC 方法，服务器找到对应的处理器执行，把结果原路返回。每个请求都有唯一的 ID，响应会带上同一个 ID，这样客户端就知道哪个响应对应哪个请求——哪怕多个请求同时在飞，也不会搞混。

**事件推送（Push）**：服务器不只是被动应答。当 Agent 产生了新的消息、权限请求、状态变更等事件时，服务器会主动向客户端推送。这就是 WebSocket 的杀手锏——不用客户端反复问"好了没"，服务器自己会说"我好了"。

## 掉线了怎么办？事件缓冲和重连回放

网络不会永远稳定。Wi-Fi 闪断、手机切后台、服务器重启——连接随时可能断。Craft Agents 为此设计了一套可靠投递机制：

**事件缓冲（Event Buffer）**：服务器为每个客户端维护一个环形缓冲区（Ring Buffer），最多保留 500 条最近的事件，超过 30 秒的旧事件会被淘汰。客户端在线时，事件正常发送的同时也写入缓冲区；客户端离线后，缓冲区继续暂存新事件，最多保留 60 秒——给重连留一个"窗口期"。

**重连回放（Reconnect Replay）**：客户端断线后，会自动以指数退避（Exponential Backoff）策略尝试重连——第一次等 1 秒，第二次 2 秒，第三次 4 秒……最长等 30 秒。重连时，客户端告诉服务器"我上次收到第 N 条消息"，服务器从缓冲区里找出第 N+1 条之后的所有事件，按顺序重放给客户端——一条都不会少。

当然，如果断线太久，缓冲区里的事件已经过期被淘汰了，服务器会告诉客户端"你错过了太多，需要做一次全量刷新"——这就是"过期"（Stale）状态，客户端收到后会重新拉取完整数据。

## 客户端能力：我擅长什么

不是所有客户端都能做所有事。桌面端能弹出原生文件对话框，Web 端不行；Electron 客户端能打开新窗口，浏览器里的客户端做不到。

所以客户端在握手时，会告诉服务器自己有哪些能力（Client Capabilities），比如：能打开链接、能选择文件、能弹出对话框等。服务器需要调用客户端的功能时（没错，方向可以反过来），会先检查客户端是否声明了对应的能力。如果客户端不支持，服务器就不会调用，避免白忙活。

## RPC 通道：电话分机号

最后来说说"按哪个按钮"的问题。

RPC 方法名在 Craft Agents 中叫做**通道（Channel）**，定义在 `@craft-agent/shared/protocol` 的 `RPC_CHANNELS` 中。通道名采用"命名空间:方法名"的格式，就像公司的电话分机号——拨 `sessions:get` 找会话查询，拨 `sessions:sendMessage` 发消息，拨 `sources:list` 查数据源，分工明确，不会串线。

主要的通道命名空间包括：

| 命名空间 | 用途 | 举例 |
|---------|------|------|
| `sessions` | 会话管理 | `sessions:get`、`sessions:sendMessage`、`sessions:cancel` |
| `server` | 服务器状态 | `server:getStatus`、`server:shuttingDown` |
| `sources` | 数据源管理 | `sources:get`、`sources:startOAuth` |
| `file` | 文件操作 | `file:read`、`file:openDialog` |
| `workspace` | 工作区配置 | `workspace:getPermissions`、`workspaceSettings:get` |
| `window` | 窗口管理 | `window:openWorkspace`、`window:close` |

服务器在握手响应中，会把自己注册了哪些通道告诉客户端。客户端调用前可以先检查 `isChannelAvailable()`，避免调用一个不存在的通道。
