# 4.1 为什么要单独运行？进程隔离的意义

上一章我们了解了权限系统——它决定了 Agent 能做什么、不能做什么。但权限只是故事的一半：Agent 在**哪里**运行，同样至关重要。

想象一家医院的病房走廊。如果所有病人都住在一个大开间里，一个人咳嗽，全屋子的人都得跟着吃药；一个人半夜梦游，所有人都会被吵醒。所以医院设计了**隔离病房**——每个房间独立通风、独立照明，里面出了问题，不会波及隔壁。

Craft Agents 里的 Pi Agent Server 就是这样一间隔离病房。它不跟主程序住在同一个"房间"里，而是作为独立的子进程（Subprocess）运行。这一节我们就来聊聊：为什么要费这个劲？

## 什么是进程？

在理解"隔离"之前，先搞清楚"进程"是什么。

进程（Process）就是一个独立运行的程序，拥有自己的内存空间。打个比方：进程就像公寓楼里的一户人家——有自己的房间和家具，和其他住户互不干扰。你家的冰箱里放了什么，隔壁邻居看不到，也碰不着。

在 Craft Agents 中，有两个重要的进程：

- **主进程（Main Process）**：运行你的应用程序界面、处理用户交互、管理工具调用——这就是"公寓楼"本身
- **子进程（Subprocess）**：运行 Pi Agent Server——这就是"租了隔壁房间"的住客

## 为什么需要进程隔离？

如果 Pi Agent 只是个普通模块，直接在主进程里跑不就行了？还真不行。原因有四个：

### 1. 崩溃隔离（Crash Isolation）

AI 引擎不是完美的。Pi SDK 在处理大模型响应时，可能因为格式异常、网络超时、或者模型输出了意料之外的内容而崩溃。如果它和主程序跑在同一个进程里，一个崩溃，整个应用就跟着倒下——就像大开间里一个病人发烧，所有人都得跟着隔离。

但在子进程里运行就不一样了。Pi SDK 崩溃了？主进程会收到一个"子进程意外退出"的通知，然后从容地处理——也许是重启子进程，也许是给用户一个友好的提示。无论如何，你的界面不会卡死，用户数据不会丢失。

### 2. 资源管理（Resource Management）

大模型推理是吃资源的大户。一个请求可能占用几百 MB 内存，消耗大量 CPU 进行 JSON 解析和上下文管理。如果这些全都发生在主进程里，用户会明显感受到界面卡顿——点击按钮没反应、滚动掉帧、动画变幻灯片。

子进程可以独立分配 CPU 和内存资源。主进程专注界面渲染，子进程专注模型推理，各干各的，互不抢占。操作系统的调度器也会更智能地分配时间片，因为两个进程的优先级可以独立设置。

### 3. 安全隔离（Security Isolation）

这是最容易被忽略、但最重要的一点。

主进程里往往存着各种敏感信息：API 密钥、数据库连接串、用户凭证……这些信息通常以环境变量（Environment Variable）的形式存在。如果子进程和主进程共享同一个内存空间，AI 模型生成的代码或工具调用理论上可以读取到这些变量——这可不是闹着玩的。

Craft Agents 在启动子进程时，会**过滤环境变量**。像 `ANTHROPIC_API_KEY`、`AWS_SECRET_ACCESS_KEY` 这类敏感变量，会被主动屏蔽，不会传递给子进程。这就像隔离病房有独立的通风系统——主程序里的"密码味道"不会被吹到子进程的房间里去。

### 4. 多 Provider 支持（Multi-Provider Support）

不同的 AI 提供商（Provider）需要不同的运行环境。比如 OpenAI 的 SDK 需要一套环境变量，Anthropic 的需要另一套，本地模型又需要 GPU 驱动配置。如果所有东西都塞在一个进程里，环境配置会互相冲突。

子进程可以按需配置：每个 Provider 对应的 Pi Agent 实例，可以拥有自己独立的环境变量和依赖项，互不干扰。

## 和 ClaudeAgent 的对比

说到这里，你可能会想：ClaudeAgent 怎么不需要子进程？

没错，这是一个很好的问题。ClaudeAgent 和 PiAgent 采用了完全不同的运行策略：

| | ClaudeAgent | PiAgent |
|---|---|---|
| 运行方式 | 进程内运行（In-Process） | 子进程运行（Subprocess） |
| 生活类比 | 和自己人住一起 | 租了隔壁房间 |
| 优势 | 通信快、延迟低 | 隔离性好、安全性高 |
| 代价 | 崩溃风险传导 | 进程间通信开销 |

ClaudeAgent 之所以可以在进程内运行，是因为它直接调用 Anthropic 的官方 SDK，行为相对可控，崩溃风险较低——就像一个靠谱的室友，不会半夜搞事情。而 PiAgent 面对的是第三方的 Pi SDK，行为更不可预测，所以需要隔离——就像一个你不太了解的新租客，先让他住隔壁观察一段时间。

## 子进程的生命周期

子进程不是一启动就一直在那跑着的。Craft Agents 采用了**懒惰启动**（Lazy Start）策略：

1. **未启动**：应用打开时，子进程还不存在，不占任何资源
2. **第一次对话**：用户发起第一次需要 PiAgent 的对话时，主进程才启动子进程——该花钱的时候再花钱
3. **运行中**：子进程保持运行，处理后续的对话请求
4. **用完后关闭**：当不再需要 PiAgent 时，子进程被优雅关闭，释放资源

这种"按需启动、用完即走"的模式，既节省了资源，又保证了用户体验。

## 小结

进程隔离听起来是个技术细节，但它解决的是真实的问题：崩溃不传染、资源不抢占、密码不泄露、配置不冲突。就像医院为什么要建隔离病房——不是因为喜欢多花钱，而是因为出了问题真的会出大事。
