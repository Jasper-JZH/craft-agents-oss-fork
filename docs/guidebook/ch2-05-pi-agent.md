# 2.5 PiAgent——Pi SDK 为什么住在另一个房间？

上一节我们看到了 ClaudeAgent 在主进程内直接调用 SDK，像在自己家里干活一样方便。但 PiAgent 的做法截然不同——它不在自己家里干活，而是在隔壁租了个工作室，通过"传纸条"来沟通。为什么？答案藏在一个叫 Subprocess（子进程）的概念里。

## 老板和外包团队

想象你是一家公司的老板。你有两种用人方式：一是让员工直接坐在你旁边的工位上，有事当面说（这是 ClaudeAgent 的做法）；二是雇一个外包团队，他们在自己的办公室里办公，你们之间靠邮件和快递沟通。

PiAgent 选择了第二种方式。主程序（Main Process）是"老板"，它启动一个独立运行的 Subprocess（子进程）作为"外包团队"。这个子进程里跑着 Pi SDK 的服务端（`pi-agent-server`），负责和 AI 模型对话、管理会话状态、处理工具调用。

为什么要这么麻烦？两个关键好处：

- **崩溃隔离**：外包团队搞砸了，不会连累老板。子进程崩溃了，主程序还能活着，可以重新启动一个新的子进程继续工作。如果 ClaudeAgent 的 SDK 抛出未捕获的异常，整个主进程都可能跟着挂掉——毕竟它们住在同一间屋子里。
- **工具多样性**：外包团队可以用自己熟悉的工具。Pi SDK 原生支持十几种 AI Provider（提供者），包括 Anthropic、OpenAI、OpenAI Codex（ChatGPT Plus）、GitHub Copilot、Google、Amazon Bedrock、Azure OpenAI、OpenRouter 以及 Custom Endpoint（自定义端点）等。这些不同厂家的接入方式差异很大，放在独立进程里更干净。

## JSONL over stdio——传纸条的规矩

老板和外包团队之间不能当面说话，得定一套"传纸条"的规矩。这套规矩叫 JSONL over stdio（通过标准输入输出传输 JSON 行格式）。

JSONL（JSON Lines）的规则很简单：**一行就是一个 JSON 消息**。主进程通过子进程的 stdin（标准输入）把命令写进去，子进程通过 stdout（标准输出）把结果写回来。每条消息都是一个完整的 JSON 对象，独占一行，就像一张折好的纸条，上面写着一条完整的指令或回复。

```typescript
// 主进程发送命令
this.subprocess.stdin.write(JSON.stringify(cmd) + '\n');

// 子进程逐行读取
this.readline.on('line', (line: string) => {
  const msg = JSON.parse(line);
  // 根据 msg.type 分发处理
});
```

这种通信方式的优点是简单可靠——不需要额外的网络端口，不需要复杂的序列化框架，只要进程还活着，纸条就能送达。

## 五步开机：子进程启动流程

外包团队不是一来就能干活的，得先办好入职手续。PiAgent 的子进程启动分为五个阶段：

1. **spawn（雇佣）**：老板调用 `spawn()` 启动子进程，就像和外包团队签了合同。此时子进程已经跑起来了，但还没拿到任何工作资料。

2. **init（发工作手册）**：老板通过 stdin 发送一条 `init` 消息，里面包含 API Key、模型名称、工作目录、会话路径、认证方式等所有必要信息。这就像给外包团队发了一本工作手册，告诉他们"用谁的账号登录、用什么模型、在哪个目录干活"。

3. **ready（准备就绪）**：子进程收到 init 后完成初始化，回一条 `ready` 消息。此时外包团队已经在工位上坐好了，随时可以接活。

4. **register_tools（登记工具）**：老板告诉外包团队"你可以用这些工具"。包括会话级工具（如 SubmitPlan、call_llm）和 MCP Source 工具等。这些工具的定义（名称、参数、描述）通过 `register_tools` 消息发送过去，让 AI 模型知道它有哪些工具可调用。

5. **prompt（派任务）**：一切就绪后，老板发送 `prompt` 消息，里面包含用户的输入和系统提示词。外包团队正式开工。

## 代理工具：打印文件得寄回总部

外包团队需要用工具时，有趣的事情来了——他们并不自己执行，而是发一条 `tool_execute_request` 消息给老板，请老板代为执行。

为什么要这样？因为很多工具（比如 MCP Source 工具、会话级工具）的运行环境在主进程这边，子进程那边没有上下文。就像外包员工需要打印一份文件，但他们办公室没有打印机——只能把文件寄给总部，总部打印完再把结果寄回来。

流程是这样的：

1. AI 模型决定调用某个工具
2. 子进程发送 `tool_execute_request`（包含请求 ID、工具名称和参数）
3. 主进程收到请求，执行工具，得到结果
4. 主进程回送 `tool_execute_response`（包含请求 ID 和执行结果）
5. 子进程把结果交给 AI 模型，继续对话

在源码中，`handleToolExecuteRequest()` 方法负责接收请求并路由到正确的工具处理器，`routeToolCall()` 则根据工具名称把调用分发到对应的执行逻辑。

## 权限检查也是代理的

工具执行之前还有一道关卡：权限检查。这件事也是外包团队交给老板办的。

子进程在执行工具前，先发一条 `pre_tool_use_request` 消息，问老板"这个操作允许吗？"。老板运行集中式权限检查流水线（`runPreToolUseChecks`），检查当前权限模式、工作目录约束、Source 激活状态等，然后回复 `pre_tool_use_response`，告诉子进程三种结果之一：

- **allow**（允许）：放心执行
- **modify**（修改）：参数需要调整后再执行
- **block**（阻止）：这个操作不允许，并附带拒绝原因

在 `ask` 模式下，如果权限检查需要用户确认，主进程还会弹出提示让用户决定，然后把用户的选择送回子进程。

## 原生高级功能

Pi SDK 还自带几个让工作更顺畅的高级功能：

**Steer（流中转向）**：AI 正在回复时，你可以插入一条新指令，就像导航过程中临时改了目的地。调用 `redirect()` 方法会发送一条 `steer` 消息给子进程，AI 会在当前工具执行完毕后读取新指令，跳过剩余的排队工具，带着完整上下文继续工作。不需要中断重来，一气呵成。

**Compact（上下文压缩）**：对话太长时，AI 的"记忆"会超出模型的上下文窗口。Pi SDK 支持自动压缩——把旧的对话内容总结成一段摘要，保留关键信息，释放 Token 空间。PiAgent 在初始化时会显式开启自动压缩，你也可以手动通过 `/compact` 命令触发。压缩完成后，你会看到一条提示，告诉你节省了多少 Token。

**Token 自动刷新**：使用 OAuth 登录的方式（如 ChatGPT Plus、GitHub Copilot），访问令牌（Access Token）有过期时间。PiAgent 检测到 401 未授权错误时，会自动用刷新令牌（Refresh Token）换取新的访问令牌，然后通过 `token_update` 消息推送给子进程——整个过程对你透明，不需要重新登录。为了防止多个 PiAgent 实例同时刷新同一个连接的令牌，还加了全局互斥锁，确保同一时刻只有一个刷新操作在跑。
