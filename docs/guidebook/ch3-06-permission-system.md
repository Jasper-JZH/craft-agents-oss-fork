# 3.6 权限系统——谁在守护安全？

上一节我们了解了 call_llm 工具，让 Agent 能够"请教同事"来完成超出自身能力的任务。但问题来了：如果 Agent 可以自由调用工具、读写文件、执行命令，万一它手滑删了重要文件怎么办？谁来守卫这扇门？

## 门禁系统：Agent 的安全关卡

想象你走进一家公司的大楼。前台给你发了一张门禁卡——大厅和走廊随便逛，但进服务器机房得刷卡，进 CEO 办公室则需要特殊审批。每个区域的准入规则不同，层层设防。

Agent 的权限系统（Permission System）也是这样。它是一套门禁机制，决定 Agent 在每一步操作前"能不能做"以及"要不要问人"。没有权限系统的 Agent，就像把大楼钥匙全交给一个实习生——不是不信任，而是太危险。

## 三级权限模式（Permission Mode）

权限系统最核心的概念是三种模式，对应三种不同级别的"门禁卡"。

### safe 模式——访客证，只看不碰

safe 模式就像一张访客证：你可以四处参观，但任何"动手"的行为都被禁止。

Agent 在 safe 模式下只能执行**只读操作**：

- 允许：Read（读文件）、Glob（搜索文件名）、Grep（搜索文件内容）、WebFetch（获取网页）、WebSearch（搜索网页）、只读 Bash 命令（比如 `ls`、`cat`）
- 禁止：Write（写文件）、Edit（编辑文件）、Bash 写操作（比如 `rm`、`mkdir`）、MCP 写操作

这就像博物馆的参观者——你可以看展品，但摸一下警报就响。safe 模式适合你只想让 Agent 帮你分析代码、搜索信息，而不希望它改动任何东西的场景。

### ask 模式——临时通行证，做事先报告

ask 模式像一张临时通行证：你可以进入各个区域，但每次做敏感操作前都要向管理员汇报。

在 ask 模式下，所有操作原则上都允许，但涉及**写入文件、执行命令、修改数据**等危险操作时，系统会弹出一个确认窗口，让你决定是否放行。这就像你每次想进服务器机房，保安都会问你："确认要进去吗？"

有个贴心的细节：你可以点击"总是允许"来记住某些命令。比如你信任 Agent 执行 `npm test`，点一次"总是允许"后，下次再跑测试就不用手动确认了——相当于给这条特定通道办了张快速通行证。

### allow-all 模式——管理员卡，畅通无阻

allow-all 模式就是一张管理员卡：所有门禁全部打开，Agent 想做什么直接做，不再询问。

这种模式的适用场景是你**完全信任 Agent**——比如你在跑一个已经验证过的自动化流水线，或者你在做快速原型不想被反复打断。但就像管理员卡不能随便发人一样，allow-all 模式也要谨慎使用。一旦开了这个口子，Agent 误操作就没有任何拦截了。

## PreToolUse 管道：六步安检流程

光有"门禁卡级别"还不够。每次 Agent 要使用工具时，还要经过一道安检流程——这就是 PreToolUse Pipeline（工具使用前管道）。它像机场安检一样，分六步逐一检查：

1. **权限模式检查**——你有权限进入这个区域吗？先看当前是 safe、ask 还是 allow-all，直接判断操作是否被允许。

2. **Source 阻断**——这个外部服务已连接吗？如果工具需要访问某个 MCP 数据源，但该数据源没有被配置或已断开，直接拦截。

3. **前置条件检查**——你读过操作指南了吗？某些工具要求先执行前置步骤（比如 Edit 之前必须先 Read），没满足条件就拒绝。

4. **call_llm / spawn_session 拦截**——特殊工具走特殊通道。上一节介绍的 call_llm 和 spawn_session 这类"跨 Agent"工具，有独立的权限逻辑，不走普通通道。

5. **输入变换**——路径标准化、参数校验。把相对路径转成绝对路径，把不合法的参数挡在门外——就像安检时把液体倒进标准容器一样。

6. **Ask 模式提示决策**——需要弹窗问用户吗？如果当前是 ask 模式且操作属于危险操作，就在这里弹出确认窗口。

这六步串行执行，任何一步不通过，工具调用就会被拒绝。就像安检流水线——任何一步发现问题，旅客就不能登机。

## 自定义权限：分区管理

三级模式是通用方案，但有时候你需要更精细的控制——就像门禁系统支持"分区管理"：研发部的人能进实验室但进不了财务室，财务部反过来。

用户可以通过 JSON 配置文件自定义权限规则，比如：

- 允许哪些 Bash 命令（只允许 `git status` 和 `npm test`）
- 允许哪些 MCP 工具（只允许只读类工具）
- 允许写入哪些路径（只能在 `/tmp` 目录下写文件）

更灵活的是，权限配置可以**分级叠加**：

1. **应用级默认**——全局最低权限，所有人默认遵守
2. **工作区级**——针对当前项目的额外规则，覆盖应用级默认
3. **Source 级**——针对某个 MCP 数据源的特殊规则，最细粒度

这就像公司的门禁管理：大楼有统一规定（应用级），每个部门可以追加自己的规矩（工作区级），某个具体房间还能再设额外限制（Source 级）。规则从粗到细，越来越精准。
