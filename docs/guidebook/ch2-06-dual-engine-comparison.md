# 2.6 双引擎对比——什么时候用哪个？

想象你开了一家翻译公司。你有一位金牌翻译住在公司楼上，随叫随到，沟通零延迟——但只精通英语；另一位是远程合作的自由译者，通过邮件交接，稍微慢一点，但精通十几种语言。你会让谁翻英文合同？谁翻日文邮件？答案不言自明。

前两节我们分别认识了 ClaudeAgent 和 PiAgent 两种引擎，它们就像这两位译者：各有专长，互补而非重复。本节就把它们放在一起，看看到底什么时候该用谁。

## 一张表看懂核心差异

| 维度 | ClaudeAgent | PiAgent | 通俗解释 |
|------|-------------|---------|----------|
| SDK 位置 | 进程内（In-process） | 子进程（Subprocess） | 一个在自己家干活，一个租了隔壁工作室 |
| 通信方式 | `query()` 的 AsyncIterable | JSONL over stdio | 面对面说话 vs 传纸条 |
| 会话工具 | 进程内执行 | 代理执行（Proxy） | 自己动手 vs 请人代劳 |
| 权限拦截 | SDK hooks.PreToolUse | 子进程请求 → 主进程响应 | 门口站岗 vs 打电话问总部 |
| 认证注入 | `process.env` 环境变量 | `init` 命令 + `token_update` | 加好油再出发 vs 运行中加油 |
| 流中转向 | `additionalContext` 注入 | 原生 `steer()` | 塞纸条 vs 内置导航 |
| 上下文压缩 | SDK 内部处理 | 显式 `compact` RPC | 自动 vs 手动 |
| Provider 支持 | 仅 Anthropic | 10+ Provider | 专用 vs 通用 |

逐行看下来，你会发现一个清晰的分野：ClaudeAgent 走的是"紧耦合"路线——一切都在同一进程内完成，通信零开销，权限就地拦截，压缩自动完成；PiAgent 走的是"松耦合"路线——通过子进程和标准输入输出通信，每次交互都经过序列化，但也因此获得了进程隔离的安全性和多 Provider 的灵活性。

打个更具体的比方：ClaudeAgent 像一位住在你家里的厨师，想加调料直接喊一声就行；PiAgent 像一位外卖大厨，你需要通过菜单下单，但他能做的菜系多得多，而且万一厨房着火也不会烧到你家。

## 什么时候用 ClaudeAgent？

**你只用 Claude，想要最原生体验。** 如果你整个项目只用 Anthropic 的模型，ClaudeAgent 就是最直接的选择。它直接调用 Anthropic SDK，没有中间层，没有序列化开销，API 的每一个特性——从流式输出到 `additionalContext` 注入——都能第一时间用上。就像直接用母语聊天，不用经过翻译。

**你追求最快响应速度。** 进程内调用意味着零通信开销。没有 JSON 序列化、没有标准输入输出的读写、没有子进程启动的等待。在需要高频交互的场景下（比如实时对话、逐 token 流式输出），这种差异是可以感知的。

## 什么时候用 PiAgent？

**你想用 Google Gemini、ChatGPT、GitHub Copilot 等。** 这是 PiAgent 最核心的优势。通过 Pi（一个通用 AI 编程助手），它支持 10 种以上的 AI Provider，包括 Google、OpenAI、GitHub 等主流平台。如果你的用户群体使用不同的 AI 服务，PiAgent 让你一份代码通吃所有。

**你想让 Agent 支持多种 AI，用户可以随时切换。** 想象你的产品是一个代码助手，有的用户订阅了 ChatGPT Plus，有的用 Gemini，有的用 Claude。你不可能为每个 Provider 写一套 Agent 逻辑——PiAgent 让你写一次，跑在所有 Provider 上。

**你需要进程隔离的安全性。** 子进程意味着 Agent 的代码运行在独立的进程中。如果它崩溃了、内存泄漏了、甚至被恶意代码攻击了，主进程不受影响。这就像你不会让装修工人住在自家客厅——让他们在隔壁工作室干活，更安全。

## 为什么不全用一种？

你可能想问：既然 PiAgent 支持那么多 Provider，能不能干脆只用它，把 ClaudeAgent 扔了？

技术上当然可以，但你会在 Claude 这个最常用的场景下付出不必要的代价：每一次调用都多了一层序列化和跨进程通信，每一次权限检查都要绕一圈，上下文压缩也从自动变成了手动。这就好比你明明有楼上的金牌翻译，非要把英文合同也寄给远程译者——不是不行，只是没必要的折损。

反过来，如果只用 ClaudeAgent，你的 Agent 就被锁定在了 Anthropic 的生态里。明天用户说"我想换 Gemini"，你只能重写整个后端。

所以答案是：**互补而非重复**。ClaudeAgent 专注做好 Claude 的原生体验，PiAgent 专注做好多 Provider 的通用支持。两者各司其职，让用户按需选择。

## 从用户视角看：毫无区别

最后说一个关键点：无论底层用的是 ClaudeAgent 还是 PiAgent，对使用你产品的人来说，体验是完全一致的。

他们不会看到 AsyncIterable 还是 JSONL，不会感知到进程内还是子进程。他们只会在设置页面看到一个下拉菜单——"选择 AI 模型"——选 Claude 就是 ClaudeAgent 在跑，选 Gemini 就是 PiAgent 在跑。切换引擎对用户来说，就是换个选项的事。

这也是 Craft Agents 架构设计最精妙的地方：**底层双引擎，上层一张皮**。复杂留给框架，简单留给用户。
